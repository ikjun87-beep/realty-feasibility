# PRD — 사업수지 통합 보고서 (realty-feasibility × cashflow-tool)

> **한 줄 정의:** cashflow-tool 엔진이 산출한 **수입-지출-현금흐름(사업수지)** 결과를, realty-feasibility의 **발표형 보고서 디자인**으로 출력하는 통합 보고서 시스템. 최종형은 *별첨자료(토지·설계) 입력 → 엔진 자동검토 → 발표형 종합 보고서 PDF 회신*.
>
> 문서버전 v0.1 · 작성 2026-06-28 · 작성자 Claude(자가검증 진행) · 사용자 확인 패스(위임)

---

## 0. 배경 — 두 프로젝트의 관계

| | **realty-feasibility** (당하동) | **cashflow-tool** (양산 외) |
|---|---|---|
| 정체 | 발표형 **보고서 디자인 시스템**(가로 16:9 HTML→PDF) | 부동산개발 **사업수지 검토 엔진**(단일 HTML) |
| 강점 | Pretendard·다방 규율·다크헤더·골드, 위치도(VWorld)·표·KPI 디자인 | "0원 일치" 검증, 수입·지출 40항목·CF·이익률·BEP·신탁구조·부가세 |
| 한계 | **수입-지출(사업수지) 의도적 제외** | 보고서 출력이 실무 표 위주(발표용 디자인 약함) |

**통찰:** 두 프로젝트는 정확히 상호보완. 당하동 보고서에서 빠졌던 **수입-지출(사업수지)** 를 cashflow-tool 엔진이 법령·약관 기준으로 정확 산출한다. 본 작업 = **cashflow-tool Phase 3(종합 보고서 생성)을 realty-feasibility 디자인 언어로 구현**.

## 1. 목표 · 비전

### 1.1 최종 비전 (cashflow-tool 진행현황 §1과 정합)
```
[홈페이지 web/request.html] 토지·설계개요 별첨 입력
        ↓ ParseInput 파서
[숨김 iframe: 사업수지검토_표준툴.html] 엔진 계산(잔차0·멱등)
        ↓ RESULT 객체
[통합 보고서 생성기] realty-feasibility 디자인 적용
        ↓
[발표형 종합 보고서 + 발표자료 PDF] 즉시 회신/다운로드
```

### 1.2 본 PRD의 확정 범위 (이번 사이클)
1. **수입-지출(사업수지) 섹션을 보고서에 신설** — 엔진 출력을 발표형 슬라이드로.
2. **양산 물금 모델로 end-to-end 예시 1부** 작성(엔진 실행값 기반, 잔차0 배지).
3. **2인 전문가 리뷰 루프**로 98/100까지 품질 수렴.

### 1.3 불변 원칙 (양 프로젝트 규칙 승계)
- **엔진 진실식(불변):** 사업이익 = 총분양수입 − 지출(VAT포함) = CF 누적시재, **잔차 0·멱등·무오류**. (cashflow-tool §11) — 특정 목표값에 맞추지 않는다.
- **INPUT 단가 보호:** `_v23Locked` 등 확정 입력단가는 변경 금지(cashflow-tool CLAUDE.md §5). OUTPUT 사업이익은 동결 대상 아님.
- **위치도 핀 정확성 필수:** 실주소 지오코딩(VWorld), 추정 금지(realty-feasibility 기준).
- **사업성 가부 판정 미내장:** 지표만 제시, 가부는 심사자 몫(cashflow-tool 결정 §3).

## 3. 워크플로우 정리 (지금까지의 모든 작업)

### 3.1 realty-feasibility 측 (보고서 툴) — 확립된 워크플로우
1. **SSOT=HTML, PDF는 파생물.** 수정은 HTML, PDF 재생성.
2. **렌더:** Windows Chrome 헤드리스(한글경로 회피 ASCII 빌드폴더). *폴백:* WSL 인터롭 장애 시 **Linux Chromium**(`~/.cache/ms-playwright/.../chrome`) — 검증됨.
3. **검증:** 슬라이드별 고해상 스크린샷(`section:nth-of-type(N)` 격리 렌더).
4. **디자인 규율(다방):** Pretendard 단일 패밀리, box-shadow 0·1px 헤어라인, 골드=브랜드/회색=콘텐츠 3계층, 다크 그래파이트 헤더 밴드(표지 톤 전 페이지).
5. **위치도:** VWorld 지번 지오코더(키 URL방식, WebFetch 호출 가능) → 좌표 확정 → Leaflet 정적 렌더.
6. **표지 이미지:** 오리지널 SVG 황혼 렌더(저작권 클린, 외부사진 미사용).

### 3.2 cashflow-tool 측 (엔진) — 확립된 워크플로우
1. **입력:** 사업장 JSON(토지·건축·세대·일정·금융·지출 40항목) 또는 web 폼→파서.
2. **호출 순서(폐쇄조건):** `init → loadStandard → calculate → autoFillExpenses → calculate → updateCashflow → updateFinalReport`.
3. **계산:** 외부루프(사업이익 고정점)×내부루프(CF 수렴, |Δ|<0.5천원), 멱등.
4. **검증:** `node _verify/run_headless.js`(양산 잔차0·멱등), `audit_*.js` 회귀.
5. **신탁구조 자동판정:** 토지비/공사비 비율 ≥10%(공격적 15%) → 하이브리드.
6. **정확성 방어:** 40항목 전부 법령·약관·표준요율로 역산 가능(`검증_지침부합_양산.md`). 원본 엑셀 대비 신탁보수·금융비 완전반영 → 보수적·정확(`방어근거_원본대비_정확성.md`).

### 3.3 통합 파이프라인 (신규)
```
별첨(토지·설계) ─▶ parse_input.js ─▶ 엔진(run_headless/iframe) ─▶ RESULT(JSON)
                                                              │
realty-feasibility 디자인 토큰 ◀──────────────────────────────┘
        │
        ▼
통합 보고서 HTML(수지 섹션 포함) ─▶ Chrome/Linux Chromium ─▶ 종합보고서 PDF
        │
        ▼
2인 전문가 리뷰 루프(≥98/100) ─▶ 확정본
```

## 4. 적용가능 항목 + 적용법 (엔진 출력 → 보고서 매핑)

> 엔진 RESULT 객체(`사업수지검토_표준툴.html` line 1159~) → 발표형 슬라이드. **값은 엔진 실행으로 취득**(하드코딩 금지).

| # | 엔진 출력 | 보고서 표현(신규 슬라이드/요소) | 디자인 |
|---|---|---|---|
| 1 | `profit`·`profitRatioClient`·`profitTrustee`·`profitRatioTrustee`·`bep1/2` | **사업수지 한눈에(KPI 4~6칸)**: 위탁/수탁 이익률·BEP·사업이익 | KPI 카드(골드 강조) |
| 2 | `totalPrice`·`totalMain`·`totalBalcony`·`totalVatAll` | **수입 구성표**(본분양/발코니/매출VAT/합계, 평단가) | 표 + 도넛 |
| 3 | `catTotals`(8대) + `totalExpenseVatIncl` | **지출 분류표(A토지~H금융)** 금액·비중% | 표 + 가로 막대 |
| 4 | `cfMonthly[]`·`cfFinalBalance`·일정 마커 | **월별 현금흐름 요약**(분기/연 집계 + 누적선) | 라인/영역 차트 |
| 5 | `trustBorrowTotal`·`trustInterestTotal`·`seniorPFInterestTotal`·`hlBorrowTotal` | **금융·신탁 구조도**(PF/신탁계정대/하이브리드 토지비대출·이자) | 구조 다이어그램 |
| 6 | 부가세 정산(매출VAT−매입VAT) | **부가세 분기정산 요약** | 미니 표 |
| 7 | 분양율 민감도(6/12/18/24/준공) | **민감도(시나리오) 표** — "진짜 위험은 안 팔릴 때" | 표 + 신호색 |
| 8 | `cfFinalBalance` 항등식·잔차 | **검증 배지**: "매출−지출=CF시재, 잔차 0·멱등" | 골드 인장 배지 |
| 9 | `trustType`·하이브리드 판정근거 | **사업구조(신탁) 설명** 1슬라이드 | 콜아웃 |

**가부 판정은 넣지 않는다**(지표만). 추정·가정은 각주로 명시(분양율 가정 등).

## 5. 통합 보고서 목차 (기존 + 신규)

> realty-feasibility 12장 디자인을 계승하되, **수지 파트(④그룹) 신설**. 양산 예시 기준 목차(당하동은 입지중심이라 수지파트가 그동안 비어 있었음 → 본 통합으로 완성).

```
A. 표지 (다크 네이비 + SVG 황혼 렌더)
B. 한눈에 보기 (사업개요 + 핵심 KPI: 이익률·BEP·사업이익·총사업비)
C. 입지·위치도 (VWorld 핀)              ← realty 자산
D. 사업·건축 개요 / 법규                  ← realty 자산
E. 분양 계획 · 추천 분양가 / 시세         ← realty 자산
─────────────  ▼ 신규 수지 파트(cashflow-tool)  ─────────────
F. 사업수지 한눈에 (수입−지출=이익=CF, 잔차0 배지)
G. 수입 구성 (분양수입·발코니·매출VAT)
H. 지출 구성 (8대 분류 + 비중)
I. 금융·신탁 구조 (PF/신탁계정대/하이브리드)
J. 월별 현금흐름 (누적선 + 일정마커)
K. 부가세·민감도 (분기정산 + 분양율 시나리오)
─────────────────────────────────────────────────
L. 종합 결론 (지표 요약, 가부 미판정)
M. 출처·면책 (엔진 산식·법령 근거·검증)
N. 마무리 (웜 에스프레소 톤)
```

## 9. 2인 전문가 리뷰 루프 (Step 5) — 핵심 품질 게이트

### 9.1 페르소나 (둘 다 공통 성격: 비판적·깐깐·정확·미적감각·엄격·신중·보수·워크홀릭·20년차)
- **심사역 A — 부동산 사업성검토 전문가(투자분석):** 수지 산식·가정의 타당성, 신탁구조·금융비·부가세·BEP·민감도의 정확성과 보수성, 추정 표기, 법령 정합. *"숫자가 틀리거나 가정이 낙관적이면 가차없이 감점."*
- **심사역 B — PPT/보고서 디자인 전문가:** 레이아웃·오와열·여백 균일·타이포·색 규율·표/차트 가독·서사 흐름·발표 적합성. *"한 픽셀의 어긋남도 지적."*

### 9.2 루브릭 (100점)
| 영역 | 배점 | 심사 |
|---|---|---|
| 수지 정확성(잔차0·산식·법령) | 30 | A |
| 가정·보수성·리스크 표기 | 15 | A |
| 투자지표·민감도 해석 | 10 | A |
| 레이아웃·오와열·여백 | 15 | B |
| 타이포·색·디자인 규율 | 10 | B |
| 표/차트 가독·정확 | 10 | B |
| 서사·발표 적합성 | 10 | B |

### 9.3 루프 절차 (무한 반복, 목표 ≥98)
```
1. 보고서 렌더(전 슬라이드 스크린샷)
2. A·B 에이전트 병렬 평가 → 각 점수 + 구체적 수정의견(파일·위치·근거)
3. 종합점수 = A합 + B합
4. if 종합 < 98 → 수정의견 반영(HTML 수정·재렌더) → 2로 회귀
   else        → 확정(잔차0 재확인 포함)
5. 수렴 안전장치: 동일 지적 3회 미해결 시 사용자에게 에스컬레이션(설계상 트레이드오프일 수 있음)
```
- **자가검증 게이트:** 매 루프 잔차0·멱등 재확인, 핀 정확, 단위 라벨 점검.

## 10. 품질 게이트 (정량)

- 엔진: 잔차 0, 멱등(run_headless PASS).
- 보고서: 리뷰 종합 ≥ 98/100.
